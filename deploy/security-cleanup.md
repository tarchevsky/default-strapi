# Очистка после компрометации (майнер Kinsing / kdevtmpfsi)

**Важно:** сначала удалить крон-задачу, иначе через минуту майнер снова скачается и запустится.

## Как майнер попадает на сервер (механизм заражения)

Серверы в облаке (в т.ч. только что созданные) постоянно сканируются ботами по всему интернету. Достаточно одного открытого уязвимого сервиса — заражение может произойти за минуты или часы после появления IP в сети.

**Типичные векторы для Kinsing и подобных майнеров:**

1. **Redis без пароля, слушает 0.0.0.0**  
   Если Redis доступен с интернета (порт 6379) и не настроен `requirepass`, бот подключается, выполняет что-то вроде:
   - `CONFIG SET dir /var/spool/cron/crontabs` и записывает задание в крон (каждую минуту `wget ... | sh`), или
   - запись в `~/.ssh/authorized_keys` для доступа по SSH.  
     После этого крон по расписанию качает скрипт с сервера злоумышленника; скрипт ставит бинарники (kdevtmpfsi, kinsing) и добавляет персистентность (крон, .bashrc).

2. **SSH: слабый пароль или утёкший ключ**  
   Подбор пароля (brute-force) или использование утёкших учётных данных. После входа злоумышленник вручную или скриптом ставит крон и майнер.

3. **Уязвимое веб-приложение**  
   RCE (remote code execution) в CMS, панели, плагине — выполнение команд на сервере и дальнейшая установка майнера.

4. **Открытый Docker API**  
   Порт 2375/2376 без TLS и аутентификации — создание контейнера с примонтированным каталогом хоста и запись в крон или автозагрузку.

**Почему «сервер создал вчера» не защищает:** боты не выбирают «старые» сервера — они сканируют все IP по портам (6379, 22, 80, 443 и т.д.). Новый VPS получает IP из пула; этот IP могли сканировать уже в первый день. Достаточно один раз поднять уязвимый Redis или попасть в подбор SSH — заражение произойдёт без участия человека.

**Что проверить у себя:** был ли на сервере Redis, слушал ли он 0.0.0.0; не использовался ли простой/повторно использованный пароль SSH; не ставились ли панели/скрипты с известными уязвимостями. В логах: `journalctl -u redis-server` (если был), `grep Accepted /var/log/auth.log` — с каких IP заходили по SSH.

## Как выявлять крон-задачи, качающие майнер

**Признаки заражения:** высокая загрузка CPU (≈100% us в `top`), процесс из `/tmp` с именем вроде `kdevtmpfsi` или `kinsing`, крон раз в минуту качает скрипт по HTTP и запускает его через `sh`.

**1. Нагрузка и процессы**

```bash
top -c          # Сортировка по CPU: кто жрёт ядра
ps aux          # Ищи процессы из /tmp (kdevtmpfsi, kinsing и т.п.)
ps aux | grep -E '/tmp/|wget|curl'   # Подозрительные бинарники и загрузки
```

**2. Крон пользователя**

```bash
crontab -l
```

Подозрительно: строки с `wget`/`curl` на неизвестный URL и передачей в `sh`/`bash`, например:
`* * * * * wget -q -O - http://... | sh`

**3. Системный крон**

```bash
cat /etc/crontab
ls -la /etc/cron.d/
for f in /etc/cron.d/*; do echo "=== $f ==="; cat "$f"; done
```

Ищи те же паттерны: `wget`, `curl`, пайп в shell, неизвестные хосты.

**4. Файлы в /tmp и /var/tmp**

```bash
ls -la /tmp
ls -la /var/tmp
```

Подозрительно: исполняемые файлы от обычного пользователя (kdevtmpfsi, kinsing, libsystem.so), папки вроде `.ICEd-unix`, конфиги (config.json).

**5. Слушающие порты**

```bash
ss -tlnp
```

Неизвестный процесс (не nginx, node, postgres и т.д.), слушающий случайный высокий порт — возможный бэкдор (kinsing часто слушает свой порт).

## 1. Удалить крон (сделать первым делом)

```bash
crontab -r
```

Проверить, что крон пуст: `crontab -l` → должно быть "no crontab for itdev3" или пусто.

## 2. Убить процессы майнера

**Как узнать PID:** в выводе `ps aux` PID — второй столбец (число после имени пользователя).

```bash
# Найти процессы и их PID
ps aux | grep -E 'kdevtmpfsi|kinsing'
```

Пример вывода (второй столбец = PID):

```
itdev3  35670  0.0  ...  /tmp/kinsing      ← PID 35670
itdev3  41927  189  ...  /tmp/kdevtmpfsi   ← PID 41927
itdev3  42107  0.0  ...  grep -E ...      ← это сам grep, не трогать
```

Убить майнеры (подставь свои PID из второго столбца):

```bash
sudo kill -9 35670 41927
```

Или одной командой убить по имени процесса:

```bash
sudo pkill -9 -f kdevtmpfsi
sudo pkill -9 -f kinsing
```

По `ss -tlnp` kinsing слушает порт (например 30564) — после kill порт освободится.

## 3. Удалить файлы майнера

```bash
sudo rm -f /tmp/kdevtmpfsi /tmp/kinsing /tmp/libsystem.so
sudo rm -rf /tmp/.ICEd-unix
sudo rm -f /var/tmp/config.json
sudo rm -rf /var/tmp/.ICEd-unix
sudo rm -f /var/tmp/systemd-logind
```

## 4. Проверить системный крон и cron.d

Не выполнять файлы — только просмотр:

```bash
cat /etc/crontab
ls -la /etc/cron.d/
cat /etc/cron.d/*
```

Если есть подозрительные задания (wget/curl на неизвестные URL) — удалить или отредактировать файлы в `/etc/cron.d/`.

## 5. Проверить автозагрузку и доступ по SSH

```bash
cat ~/.bashrc
cat ~/.profile
cat ~/.zshrc
cat ~/.ssh/authorized_keys
```

Удалить неизвестные ключи из `authorized_keys`, подозрительные строки из rc-файлов.

**Частая персистентность:** в `~/.bashrc` могут быть строки вроде:
`cd /var/tmp && nohup ./systemd-logind -c config.json >/dev/null 2>&1 &`  
Удалить их (например, отредактировать `nano ~/.bashrc` или одной командой):

```bash
sed -i '/systemd-logind -c config.json/d' ~/.bashrc
```

## 6. (По желанию) Заблокировать IP злоумышленника

```bash
sudo iptables -A OUTPUT -d 80.64.16.241 -j DROP
```

Или через UFW, если им пользуетесь.

## 7. Сменить пароль пользователя itdev3

```bash
passwd
```

## 8. Проверить, что очистка сработала

```bash
# Нет майнера в процессах
ps aux | grep -E 'kdevtmpfsi|kinsing'

# Нет лишних портов (раньше был kinsing на 30564)
ss -tlnp

# Крон пуст
crontab -l
```

После этого при необходимости перезагрузить сервер и снова проверить `top` и `ps aux`.
